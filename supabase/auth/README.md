# Домен `auth` — идентичность и роли

Отвечает на вопрос «**кто** пользуется приложением». Поездки, заказы, заработок —
это уже другие домены, которые ссылаются сюда.

## Сущности

| Таблица            | Зачем                                                              |
|--------------------|-------------------------------------------------------------------|
| `auth.users`       | Управляется самим Supabase. Логин по телефону + OTP (WhatsApp). Мы её **не трогаем**. |
| `public.profiles`  | Прикладные данные пользователя 1:1 к `auth.users` (имя, фото, телефон). |
| `public.user_roles`| Роли пользователя: `client` и/или `driver`.                       |

## Ключевые решения и почему так

### 1. Зачем отдельная `profiles`, если есть `auth.users`?
`auth.users` принадлежит Supabase — туда нельзя добавлять свои колонки и опасно
читать её напрямую из клиента. Поэтому стандартный паттерн: своя таблица
`public.profiles` с `id = auth.users.id` (1:1, `on delete cascade`). Удалили
пользователя в auth — профиль удалился автоматически.

### 2. Профиль создаётся **триггером**, а не приложением
Когда Supabase создаёт запись в `auth.users` (после подтверждения OTP), триггер
`handle_new_user` автоматически создаёт строку в `profiles` (и роль, если она
передана в метаданных при регистрации). Почему триггер, а не вставка из приложения:
- надёжно — профиль есть всегда, даже если приложение упало после регистрации;
- безопасно — функция `security definer` обходит RLS контролируемо.

### 3. Роли вынесены в отдельную таблицу `user_roles`, а не колонкой
В этом приложении **одна программа — две роли**, и один человек реально может быть
и клиентом, и водителем (водитель тоже заказывает такси). Колонка `role` это бы не
позволила. Таблица «пользователь ↔ роль» (PK `(user_id, role)`) даёт:
- одну или обе роли у одного аккаунта;
- простое «дай всех водителей» (запрос по роли);
- расширяемость без `ALTER TABLE`.

### 4. Почему пользователь **не может** сам менять `user_roles`
RLS на `user_roles` разрешает только **читать свои** роли. Вставка/изменение —
только через доверенный код (`security definer` функция или `service_role`).
Иначе клиент мог бы выдать себе любую роль. Важно: роль `driver` — это лишь «человек
зарегистрировался как водитель». Право реально брать заказы зависит от **статуса
проверки** (домен `drivers`, отдельная таблица и модерация), а не от самой роли.

### 5. Функция-помощник `public.has_role(uid, role)`
Нужна для RLS в других доменах: например, «принимать заказ может только водитель».
Сделана `security definer` + `stable`, чтобы политики могли проверять роль, не
утыкаясь в собственный RLS таблицы `user_roles`.

## Состав домена
- `tables/00_enums.sql` — тип `user_role`.
- `tables/01_profiles.sql` — таблица `profiles`, триггеры `updated_at` и `handle_new_user`.
- `tables/02_user_roles.sql` — таблица `user_roles` и функция `has_role`.
- `indexes/01_auth_indexes.sql` — индексы под частые выборки.
- `rls/01_profiles_rls.sql` — политики для `profiles`.
- `rls/02_user_roles_rls.sql` — политики для `user_roles`.
