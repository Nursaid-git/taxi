# Домен `rides` — поездки (заказы)

Сердце приложения. Хранит заказ от создания клиентом до завершения и оценки,
и связывает клиента с водителем.

## Сущности

| Таблица              | Зачем                                                            |
|----------------------|-----------------------------------------------------------------|
| `public.rides`       | Сам заказ: кто, откуда, статус, цена, тариф, оплата, времена.    |
| `public.ride_stops`  | Точки назначения по порядку (можно несколько — «+ адрес»).       |
| `public.ride_ratings`| Оценка и комментарий после поездки (от клиента и/или водителя).  |

## Жизненный цикл заказа (`ride_status`)

```
searching ──accept──▶ accepted ──arrived──▶ arrived ──start──▶ in_progress ──complete──▶ completed
    │                    │                     │
    └────────────────────┴─────────cancel──────┴────────────▶ cancelled
    │
    └──(нет водителя / таймаут)──▶ expired
```

Один enum-статус описывает заказ для **обеих** ролей — это один и тот же объект,
просто клиент и водитель видят его с разных сторон. Времена каждого перехода
пишем в отдельные колонки (`accepted_at`, `arrived_at`, …) — это нужно и для
истории, и для аналитики, и для расчёта ожидания/длительности.

## Главное архитектурное решение: чтение через RLS, запись через функции

- **Чтение** (`SELECT`) разрешаем через RLS по принадлежности: клиент видит свои
  заказы, водитель — свои (назначенные) и свободные (`searching`) заявки.
- **Запись** (создание и все переходы статуса) — **только через `SECURITY DEFINER`
  функции** (`request_ride`, `accept_ride`, `start_ride`, …).

Почему не давать клиенту прямой `UPDATE` через RLS? Потому что RLS проверяет
«можно ли тронуть строку», но **не умеет проверять переход состояния**. Через
голый UPDATE водитель мог бы перескочить `accepted → completed`, выставить себе
любую цену, «принять» уже принятый заказ или назначить себя на чужой. Функция же:
- проверяет роль (через `auth.has_role`);
- проверяет, что текущий статус допускает переход;
- меняет ровно нужные поля и ставит время.

Так бизнес-правила собраны в одном месте и их нельзя обойти.

## Кросс-доступ «клиент ↔ водитель»

В домене `auth` профили закрыты (каждый видит только свой). Но во время поездки
участники должны видеть друг друга (имя, авто). Это решаем **точечно здесь**, а не
открытием профилей всем: функции возвращают ровно нужные поля участникам активной
поездки, а RLS на `rides` пускает водителя к назначенному заказу и наоборот.

## Геоданные

Координаты храним как `double precision` (`lat`/`lng`) — этого достаточно, чтобы
нарисовать точки и маршрут. Поиск ближайших водителей («кому предложить заказ»)
позже сделаем через PostGIS или отдельный домен `driver_locations` — это уже про
матчинг, а не про хранение заказа.

## Состав домена
- `tables/00_enums.sql` — `ride_status`, `ride_class`, `payment_method`, `ride_actor`.
- `tables/01_rides.sql` — заказы + триггер `updated_at`.
- `tables/02_ride_stops.sql` — точки назначения.
- `tables/03_ride_ratings.sql` — оценки и комментарии.
- `indexes/01_rides_indexes.sql` — индексы под «мои поездки», очередь `searching`, рейтинг.
- `rls/*` — политики (только SELECT по принадлежности).
- `functions/*` — создание заказа и переходы статуса (вся запись).
