# Домен `notifications` — уведомления

Лента уведомлений (иконка-колокольчик) и токены устройств для пуш-уведомлений.
Связан с поездками: смена статуса заказа порождает уведомление участнику.

## Сущности

| Таблица                | Зачем                                                        |
|------------------------|--------------------------------------------------------------|
| `public.notifications` | Уведомления пользователя (заголовок, текст, ссылка на поездку). |
| `public.push_devices`  | Токены устройств (FCM/APNs/web) для отправки пушей.           |

## Решения

### Уведомления создаёт система, а не пользователь
Пользователь уведомления только **читает** и **помечает прочитанными**. Создавать
их через клиент нельзя (иначе можно слать себе/другим что угодно). Источники:
- **триггер на `rides`**: статус сменился → кладём уведомление участнику
  («Водитель в пути», «Поездка завершена — оцените» и т.п.);
- **бэкенд** через `create_notification` под `service_role`.

Поэтому RLS на `notifications` — только `SELECT` своих; запись закрыта.
Отметка о прочтении — функцией `mark_notifications_read` (как в чате: через
RLS-UPDATE нельзя ограничить, какие колонки меняются).

### Токены устройств — это «свои данные», RLS-CRUD
`push_devices` пользователь регистрирует/удаляет сам → полный CRUD по своим
строкам через RLS. Токен уникален (`unique`), повторная регистрация — upsert.
