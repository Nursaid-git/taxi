# Домен `drivers` — водители, анкета, проверка

Всё, что делает из обычного пользователя водителя: анкета, документы, автомобиль,
**модерация**, агрегаты (рейтинг, число поездок, баланс) и статус «на линии».

## Сущности

| Таблица                  | Зачем                                                          |
|--------------------------|----------------------------------------------------------------|
| `public.driver_profiles` | Профиль водителя 1:1 к `auth.users`: статус проверки, рейтинг, баланс, онлайн. |
| `public.driver_vehicles` | Автомобили водителя (марка, модель, цвет, госномер, фото).      |
| `public.driver_documents`| Документы (права, селфи, тех.паспорт) — каждый со своим статусом проверки. |

## Главное правило: на линию — только проверенных

Статус проверки (`verification_status`): `pending → approved | rejected`.

- Выйти на линию (`set_driver_online(true)`) можно **только при `approved`**.
- Принять заказ (`accept_ride` из домена `rides`) можно **только активному
  водителю** — для этого здесь есть `public.is_active_driver(uid)` (approved + online),
  и `accept_ride` дополнен этой проверкой.

Почему важно проверять и при онлайне, и при приёме: видеть свободные заявки
(`searching`) RLS разрешает любому водителю, поэтому непроверенный мог бы дёрнуть
`accept_ride` напрямую. Гейт стоит именно в функции приёма — обойти нельзя.

## Запись — через функции, чтение — через RLS (как в `rides`)

Тот же принцип. Особенно критично для статуса: если бы водитель мог сделать
`UPDATE driver_profiles SET status='approved'`, он бы одобрил себя сам. Поэтому:
- **поля анкеты/авто/документов** пишутся функциями `save_driver_profile`,
  `set_driver_vehicle`, `add_driver_document` — они всегда ставят статус `pending`
  (любое изменение анкеты = повторная модерация);
- **модерация** (`review_driver`) доступна **только `service_role`** (бэкенд/админка),
  обычному пользователю не выдана;
- читать можно только **свой** профиль/документы. Публичная карточка водителя
  (имя, рейтинг, авто) для показа клиенту отдаётся функцией `driver_card`.

## Агрегаты считаются триггерами, а не приложением

`rating_avg` / `rating_count`, `trips_count`, `balance` — производные данные.
Их пересчитывают триггеры на доменах-источниках:
- новая оценка в `ride_ratings` → пересчёт рейтинга водителя;
- поездка стала `completed` в `rides` → `trips_count + 1`, `balance + price_final`.

Почему триггерами: цифры всегда сходятся с фактами и их нельзя «дорисовать» из
клиента. Это денормализация ради скорости (не считать AVG на каждый показ профиля).

## Состав домена
- `tables/00_enums.sql` — `verification_status`, `document_type`.
- `tables/01_driver_profiles.sql`, `02_driver_vehicles.sql`, `03_driver_documents.sql`.
- `tables/04_driver_aggregates.sql` — триггеры рейтинга/поездок/баланса.
- `indexes/01_drivers_indexes.sql`.
- `rls/*` — чтение своего.
- `functions/01_driver_self.sql` — анкета, авто, документы, онлайн, `is_active_driver`, `driver_card`.
- `functions/02_moderation.sql` — `review_driver` (только `service_role`).

> Применять **после** `auth` и `rides` (триггеры и `accept_ride` ссылаются на оба).
