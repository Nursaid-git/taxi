# Supabase — база данных приложения такси

Здесь лежит вся схема БД: таблицы, индексы и политики безопасности (RLS).
Файлы сгруппированы **по доменам** (auth, rides, drivers, payments…), а внутри
домена — **по типу артефакта** (`tables`, `indexes`, `rls`).

```
supabase/
  auth/           ← идентичность и роли (profiles, user_roles)
  rides/          ← поездки/заказы (rides, ride_stops, ride_ratings) + функции переходов
  drivers/        ← водители, анкета, модерация, рейтинг/баланс
  chat/           ← переписка по поездке
  loyalty/        ← мили клиента (счёт, журнал, награды)
  places/         ← избранные и недавние адреса
  notifications/  ← уведомления + токены устройств
  README.md       ← этот файл
```

Внутри каждого домена: `tables/` · `indexes/` · `rls/` · `functions/` (если есть)
и свой `README.md` с объяснением решений.

> **Оплаты:** пока **только наличные** (`payment_method` по умолчанию `cash`).
> Онлайн-эквайринг/выводы не делаем — домена `payments` нет намеренно.

## Почему именно такая структура

- **По доменам, а не «одна большая schema.sql».** Приложение растёт (auth,
  поездки, водители, оплаты, чат). Когда всё в одном файле — невозможно понять,
  что к чему относится, и страшно менять. Папка на домен = понятные границы.
- **Разделение tables / indexes / rls.** Это три разных уровня:
  1. `tables` — *что хранит* база (структура, связи, инварианты).
  2. `indexes` — *как быстро* мы это читаем (производительность).
  3. `rls` — *кто что* может читать/менять (безопасность).
  Их полезно держать отдельно: индексы и политики меняются гораздо чаще, чем
  сами таблицы, и ревьюить их проще по отдельности.

## Порядок применения

Внутри домена файлы пронумерованы. Применять строго по порядку:

1. `tables/` (сначала типы/enum, потом таблицы, потом триггеры)
2. `indexes/`
3. `rls/`
4. `functions/` (если есть — RPC для записи; ссылаются на таблицы, поэтому после них)

Между доменами порядок такой (по зависимостям):

1. `auth` — на него ссылаются все через `auth.users`, `has_role`, `set_updated_at`.
2. `rides` — заказы (нужны до `drivers`: триггеры и `accept_ride` их связывают).
3. `drivers` — водители (триггеры на `rides`/`ride_ratings`, гейт в `accept_ride`).
4. `chat`, `loyalty`, `places`, `notifications` — зависят от `auth` и `rides`,
   между собой независимы (порядок не важен).

### Как применить
- В дев-режиме: скопировать содержимое файла в **SQL Editor** Supabase и выполнить.
- В проде: завести миграции `supabase migration new <name>` и вставить туда SQL
  в том же порядке. (Эти файлы — «источник правды», из которого собираются миграции.)

## Общие соглашения

- Имена — `snake_case`, единственное число для типов, множественное для таблиц.
- Время — всегда `timestamptz` (UTC), поля `created_at` / `updated_at`.
- Первичные ключи — `uuid`. Профили ссылаются на `auth.users(id)`.
- `updated_at` обновляется триггером, а не руками из приложения.

## Модель безопасности (важно)

Supabase отдаёт Postgres наружу через PostgREST: мобильное приложение ходит в БД
напрямую с `anon`/`authenticated` ключом. **Поэтому единственная реальная защита —
это RLS.** Если на таблице не включён RLS или нет политик, данные либо открыты
всем, либо недоступны. Правило: на каждой таблице в `public` **включаем RLS** и
явно описываем, кто что может. Доверенные операции (выдача роли «водитель»,
системные вставки) делаются через функции `security definer` или `service_role`,
а не через политики для обычных пользователей.
