# Домен `chat` — переписка по поездке

Сообщения между клиентом и водителем в рамках конкретного заказа.

## Сущности

| Таблица                | Зачем                                                       |
|------------------------|-------------------------------------------------------------|
| `public.chat_messages` | Сообщения, привязанные к поездке (`ride_id`).               |

## Решения

### Зачем привязывать к `ride_id`, а не к «диалогу»
Отдельная таблица `conversations` не нужна: в этом приложении переписка всегда
в контексте одной поездки. `ride_id` сам по себе и есть идентификатор диалога —
меньше сущностей, проще RLS (правила доступа берём из самого заказа).

### Кто может читать и писать
- **Читать** — участники поездки (клиент и назначенный водитель).
- **Писать** — тоже только участники, и только пока поездка активная
  (`accepted` / `arrived` / `in_progress`). До назначения водителя писать некому,
  после завершения/отмены переписку закрываем.
- Здесь сообщения отправляются **прямым INSERT через RLS** (не функцией): это не
  переход состояния, а простая запись с проверкой «я участник и поездка активна».
  Такую проверку RLS выражает сама.

### Отметки о прочтении
`read_at` проставляет функция `mark_ride_messages_read` — она помечает прочитанными
сообщения собеседника. Через RLS-UPDATE это делать неудобно (нельзя ограничить,
какие именно колонки меняются), поэтому — функция.
